# 183 User Experience

Version: v1.0

---

# Purpose

RISEN CARE は

介護記録システムではない。

AIシステムでもない。

利用者理解を支援する

「介護の知能基盤」

である。

そのため

画面ではなく

利用者（User Experience）から設計する。

---

# Design Principle

利用者は

システムを操作したいのではない。

利用者を理解したい。

画面を探したいのではない。

必要な情報へ

すぐ到達したい。

RISEN CARE は

「探す」

ではなく

「理解する」

体験を提供する。

---

# User Personas

RISEN CARE の主な利用者

・介護職員

・サービス管理責任者

・管理者

・施設長

・理事長

・看護職員

・相談員

・家族（将来）

---

# Morning Experience

## 出勤

職員は

担当利用者を開く。

最初に表示されるもの

・現在の状態

・今日の観察ポイント

・前回勤務からの変化

・申し送り事項

詳細を見る必要はない。

30秒以内に状況を理解できること。

---

# During Care

ケア中

疑問があれば

AIへ聞く。

例

```
昨日と違うことは？

最近便秘は？

夜間覚醒は？

食事量は？
```

AIは

Observation

Knowledge

Episode

Pattern

を統合して回答する。

---

# Record Review

記録確認では

長文を読む必要はない。

AIが

重要な変化

のみを整理する。

必要なら

元記録を表示する。

---

# Case Conference

ケース会議では

AIが

・最近の変化

・支援効果

・注意点

・今後の提案

を整理する。

職員は

記録を探す時間ではなく

支援を考える時間を増やす。

---

# Management

管理者は

利用者単位だけではなく

施設全体を見る。

例

```
今月転倒が多い利用者

夜間支援が増えている利用者

食事量が低下している利用者
```

AIが一覧化する。

---

# Executive

理事長は

施設運営を見る。

例

```
事故傾向

委員会対応

ヒヤリハット

身体拘束

虐待リスク

加算管理
```

AIが横断して整理する。

---

# Family Support

将来的には

家族説明も支援する。

例

```
最近の生活状況

食事

睡眠

活動

楽しみ

体調
```

専門用語ではなく

家族が理解できる文章で説明する。

---

# Search Experience

RISEN CAREでは

検索画面は主役ではない。

質問が主役である。

例

```
去年の夏は？

3年前は？

最近変わった？

以前もあった？
```

AIは

Observation

Knowledge

Episode

Pattern

Review

Medical

Care Plan

を横断して回答する。

---

# Trust

AIは

推測と事実を区別する。

根拠を表示する。

元記録へ戻れる。

AIだけを信じさせない。

---

# Success Criteria

RISEN CARE が成功した状態とは

職員が

「探す時間」

より

「利用者と向き合う時間」

の方が長いことである。

---

# Vision

RISEN CARE は

介護記録を管理するシステムではない。

利用者理解を支援する

Care Intelligence Platform

である。

その中心にあるのは

AIではなく

人の支援である。
